## [→ СЛАЙД 4: Objectives]

Передо мной стояли три группы задач.

**Основная цель** — разработать систему распознавания именованных сущностей, так называемый NER, для казахского языка. Named Entity Recognition — это задача, при которой система должна в тексте автоматически находить и классифицировать имена людей, названия организаций, географических мест, дат и других сущностей.

**Технические задачи** включали освоение PyTorch и экосистемы Hugging Face, реализацию конвейера обработки данных и создание REST API.

**Учебные цели** — понять стандарты разработки в промышленной среде, применить теорию на практике и научиться работать в команде.

## [→ СЛАЙД 6: KazNERD Dataset]

Для обучения я использовал **KazNERD** — крупнейший открытый датасет именованных сущностей казахского языка. Он создан в лаборатории ISSAI при Nazarbayev University.

Датасет содержит более **112 тысяч предложений** и аннотирован для **25 типов сущностей**: персоны, организации, географические объекты, даты, законодательные акты и многие другие. Данные размечены в формате **IOB2** — это международный стандарт CoNLL-2002.

На экране виден пример: слово «Еуразиялық» начинает сущность типа LAW и получает метку B-LAW, следующие слова продолжают её — I-LAW. Обычные слова получают метку O — Outside.

После добавления B- и I- вариантов для каждого из 25 типов плюс метка O, итого модель обучалась классифицировать **51 класс**.

## [→ СЛАЙД 8: System Architecture]

Архитектура системы выглядит так. Данные KazNERD в формате IOB2 поступают в парсер, после чего строится словарь меток. Затем токенизатор XLM-RoBERTa разбивает слова на подслова, и мы выравниваем метки IOB2 с новой разбивкой. Готовый датасет подаётся на вход трансформеру. После обучения модель оценивается и экспортируется для FastAPI.

Классификационная голова — это просто линейный слой поверх энкодера с выходом на 51 класс.

## [→ СЛАЙД 9: Data Processing Pipeline]

Разберём ключевые шаги обработки данных.

**Парсер IOB2** читает файлы построчно. Каждая строка — это пара «токен — метка». Пустая строка означает границу предложения.

**Маппинг меток** сортирует все уникальные теги и присваивает каждому числовой идентификатор. Таким образом из текстовых меток мы получаем числовые векторы, понятные нейронной сети.

---

## [→ СЛАЙД 10: Tokenization & Label Alignment]

Это — одна из ключевых технических проблем проекта. 

Казахский язык является **агглютинативным**: слова образуются путём присоединения суффиксов, поэтому одно слово может стать несколькими токенами. Например, «Еуразиялық» токенизатор разбивает на «▁Еуразия» и «лық».

Проблема: у нас одна метка на слово, а токенов может быть два и более.

Решение: мы используем метод `word_ids()`, который для каждого подслова возвращает индекс исходного слова. Для первого подслова назначаем исходную метку, для последующих — соответствующую I-метку. Специальным токенам CLS и SEP присваиваем `-100` — PyTorch игнорирует их в функции потерь.

## [→ СЛАЙД 11: Model Fine-Tuning]

Мы выбрали **XLM-RoBERTa** по нескольким причинам:

Во-первых, она предобучена на ста языках, включая казахский. Во-вторых, демонстрирует сильные кросс-языковые возможности Transfer Learning. В-третьих, модель содержит 277 миллионов параметров — это хороший баланс между качеством и вычислительными затратами.

Параметры обучения: **3 эпохи**, learning rate **2e-5**, batch size **16**, оптимизатор **AdamW**, использование **fp16 mixed precision** на GPU для ускорения. Лучший чекпоинт определялся по F1-Score на валидационной выборке.

## [→ СЛАЙД 12: Evaluation Results]

И вот — ключевой слайд. Результаты на тестовой выборке KazNERD.

Наша дообученная модель достигла:
- **Precision: 88.1%**
- **Recall: 86.5%**  
- **F1-Score: 87.3%**
- **Accuracy: 94.1%**

Лучший результат у часто встречающихся сущностей: PERSON — F1 92.4%, DATE — 91.8%, GPE — 89.7%. Сложнее всего модели даются редкие классы — LAW (79.3%), поскольку юридические формулировки в казахском тексте имеют специфичный синтаксис.

Всего на тестовом множестве — почти **70 тысяч токенов** по **51 классу**.

## [→ СЛАЙД 13: API Integration]

После обучения модель была упакована в REST API с помощью **FastAPI**.

Сервис запускается командой `uvicorn api:app` и предоставляет эндпоинт `/predict`. На вход принимается казахский текст, на выходе — JSON со списком найденных сущностей: тип, текст, уверенность модели и позиции в строке.

На слайде виден пример: текст «Нұрсұлтан Назарбаев Астанада» — модель распознаёт «Нұрсұлтан Назарбаев» как PERSON с уверенностью 96.4%, а «Астанада» — как GPE (географическое место) с 95.2%.

## [→ СЛАЙД 14: Challenges & Solutions]

В ходе работы возникло несколько сложных проблем.

**Первая** — выравнивание подслов. Казахские слова с суффиксами разбиваются на множество токенов. Решили через word_ids() с обработкой крайних случаев — некоторые классы (например NON_HUMAN) не имели I-варианта в датасете.

**Вторая** — несбалансированность классов. 80% токенов — метка O. Без специальных мер модель игнорировала редкие классы. Решили через использование macro F1 как целевой метрики обучения.

**Третья** — инфраструктурные ограничения. XLM-RoBERTa требует GPU для обучения — на CPU процесс завершался с ошибкой Out-Of-Memory. Решили, перенеся обучение на GPU.
